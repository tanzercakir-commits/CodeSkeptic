# ZeroDefect — Durum Tespiti ve Yol Haritası (Temmuz 2026)

Bu doküman, projenin bağımsız bir gözle yapılmış teknik analizi, sektördeki
konumlandırması ve ilerisi için önerilen yol haritasıdır. Buradaki tüm
"doğrulandı" ibareleri bu analiz sırasında Linux (Ubuntu 24.04, LLVM 18.1.3)
üzerinde fiilen derlenip çalıştırılarak teyit edilmiştir.

---

## 1. Durum Tespiti

### 1.1 Doğrulanan durum

| Konu | Sonuç |
|------|-------|
| Linux derlemesi | Başarılı — `cmake -DCMAKE_PREFIX_PATH=/usr/lib/llvm-18` + `llvm-18-dev libclang-18-dev libzstd-dev` |
| Test paketi | **41/41 geçti** (0.79 sn) — macOS dışında ilk kez doğrulandı |
| Smoke test | uninit-ptr, reassignment leak, exit-block leak, double-free doğru yakalandı; exit code doğru (bulgu varsa 1) |
| Guard'lı bölme (parametre) | `if (z != 0) return 100/z;` — false positive YOK (parametre Unknown kaldığı için) |

### 1.2 Projenin gerçek seviyesi

Proje "üç kurallı bir linter" değil; **genel amaçlı bir dataflow analiz
çatısının çekirdeği**. Kritik mimari kararlar doğru verilmiş:

- **DataflowEngine** (template, duck-typed Analysis, worklist + merge):
  Bir kural yığınını analiz çatısına dönüştüren şey tam olarak bu soyutlama.
  Clang Static Analyzer'ın da kavramsal çekirdeği aynıdır. Yeni bir kural
  yazmak artık "lattice tanımla + transfer yaz" seviyesine inmiş durumda.
- Katmanlı mimari (SourceManager → RuleEngine → Rules → Reporter) temiz,
  bağımlılıklar tek yönlü, test edilebilirlik yüksek.
- `todo.md`'deki dürüst sınırlama listesi (bilinen FP/FN'ler) nadir görülen
  ve çok değerli bir mühendislik alışkanlığı.
- Gerçek dünya korpusuyla (cJSON, tinyxml2) doğrulama refleksi doğru.

### 1.3 Bu analizde bulunan yeni sorunlar

**(a) Linux'ta header çözümleme hatası — doğrulandı.**
`SourceManager::processAll` içindeki koşulsuz `-isystem /usr/include`
eklemesi, macOS için yapılmış düzeltmenin yan etkisi olarak Linux'ta GCC
libstdc++'ın `#include_next <stdlib.h>` zincirini kırıyor:

```
/usr/include/c++/13/cstdlib:79:15: fatal error: 'stdlib.h' file not found
```

Include arama sırasının başına `/usr/include` eklenince `include_next`
gidecek yer bulamıyor. Linux'ta `-resource-dir` zaten yeterli; `/usr/include`
ya hiç eklenmemeli ya da platforma göre koşullu olmalı. Analiz kısmi AST ile
devam ettiği için bulgular sessizce eksiliyor (demo dosyasında double-free
bu yüzden kaçtı) — **sessiz eksik analiz, yanlış "temiz" raporu demek**;
public sürüm öncesi en kritik düzeltme bu.

**(b) DataflowEngine sabit iterasyon tavanı.**
`maxIterations = numBlocks * 4` teorik bir fixpoint garantisi değil.
Monoton transfer + sonlu yükseklikli lattice ile sabitleme zaten garantidir;
tavan yalnızca güvenlik sigortası olmalı. Şu an tavana çarpılırsa
`converged=false` set ediliyor ama **sonuçlar yine de kullanılıyor** ve
kimse uyarılmıyor — döngü-yoğun fonksiyonlarda eksik state ile raporlama
riski var. Tavanı `numBlocks × lattice yüksekliği` yap, aşılırsa o
fonksiyonu "analiz edilemedi" olarak raporla.

**(c) Kurallar arası mimari tutarsızlık.**
MemoryLeakRule_Ex doğru deseni kullanıyor: tüm değişkenler tek çarpım
lattice'inde (`map<VarDecl*, AllocState>`), tek dataflow koşusu,
`dyn_cast` tabanlı hızlı classify. UninitPointerRule_Ex ise değişken başına
ayrı CFG + ayrı koşu + statement başına 5-6 ASTMatcher çalıştırıyor —
O(değişken × statement × matcher). İkisini aynı desene (çarpım lattice +
dyn_cast) toplamak hem hız hem kod sadeliği kazandırır.

**(d) DivByZero lattice'inde Unknown'un yutuculuğu.**
`merge(Zero, Unknown) = Unknown` olduğu için şu vaka sessizce kaçıyor
(doğrulandı):

```cpp
int d = 0;
if (z > 0) d = z;
return 100 / d;   // z <= 0 yolunda d == 0 — rapor YOK
```

FP önlemek için bilinçli muhafazakârlık, ama `Zero + Unknown = MaybeZero`
(en azından Info seviyesinde) daha doğru bir sinyal olurdu. Bu, guard
analizi (aşağıda) ile birlikte ele alınmalı.

**(e) Çapraz-TU mükerrer bulgu riski.**
Diagnostics TU başına toplanıp birleştiriliyor; header içinde tanımlı bir
fonksiyon birden çok TU'da analiz edilirse aynı bulgu N kez raporlanır.
Sıralama var, tekilleştirme yok — `(file, line, rule_id)` üzerinden global
dedup gerekli.

## 2. Sektör Konumlandırması — "Benzer projeler bizi vazgeçirmesin"

Dürüst tablo:

| Araç | Yaklaşım | Güçlü yanı | Zayıf yanı |
|------|----------|-----------|------------|
| clang-tidy | AST kalıpları | Ekosistem, hız | Akış analizi çoğu check'te yok |
| Clang Static Analyzer | Sembolik yürütme, path-sensitive | Derinlik | Yavaş, genişletmesi zor, TU-içi |
| Infer (Meta) | Separation logic, interprosedürel | **Diff-aware mod** | Ağır, OCaml, bakımı azaldı |
| CodeQL | Datalog sorguları | İfade gücü | Batch/CI-ölçekli, dakikalar sürer |
| cppcheck | Kendi parser'ı | Kurulum kolaylığı | Clang AST'siz, derinlik sınırlı |

Kritik gözlem: **bu araçların hiçbiri "kodu bir ajan yazıyor, döngünün
içinde birinin semantiği doğrulaması gerekiyor" iş akışı için tasarlanmadı.**
CSA'yı path-sensitivity'de yenmeye çalışmak 20 kişi-yıllık bir savaş ve
gereksiz; boşluk motorda değil, **iş akışındaki konumda**.

Senin dile getirdiğin acı — "tamam dedik, yine hata çıktı" — sektörün şu an
en pahalı problemi: üretim maliyeti sıfıra inen kodun **doğrulama maliyeti**
darboğaz oldu. ZeroDefect'in tezi bu boşluğa oturuyor:

1. **Döngü-içi kapı (in-loop gate):** LLM bir fonksiyonu değiştirir
   değiştirmez, milisaniyeler içinde sadece o fonksiyonu (ve çağıranlarını)
   analiz eden, bulguyu **dataflow izi ile birlikte** (hatalı değerin hangi
   yoldan geldiği) makine-okur formatta döndüren bir araç. İz, LLM'in
   kendini düzeltmesi için gereken açıklamanın ta kendisi. CSA/CodeQL bu
   hıza inemez; clang-tidy bu derinliğe çıkamaz.

2. **Sözleşme katmanı (intent doğrulama):** Genel UB avcılığı "evrensel"
   hataları bulur ama senin sorunun **niyet** semantiği. Hafif bir
   annotation dili (sahiplik, nonnull, aralık) → LLM kodu VE sözleşmeyi
   üretir → ZeroDefect kodu sözleşmeye karşı doğrular. "Semantik debug"
   açığını doğrudan hedefleyen budur.

3. **Diff-farkında semantik regresyon:** Her değişiklikte fonksiyon
   özetlerini (nullability, sahiplik, sıfır-olabilirlik) öncesi/sonrası
   karşılaştır; çağıranların güvendiği bir özellik sessizce değiştiyse
   alarm ver. Infer'in Meta'daki katil özelliği buydu; açık ekosistemde
   iyi yapan yok. "Tamam dedik, sonra bozuldu" tam olarak bir semantik
   regresyon problemidir.

İki yılda sıfırdan LALR parser bitiren disiplin bu iş için doğru profil:
analiz çatıları da aynı sınıf derin altyapı zanaatı — kısa yolu yok,
ama her katman bir sonrakinin üzerine birikir ve rakiplerin ataleti
(ağır, genel amaçlı, döngü-dışı) bizim çevikliğimizdir.

## 3. Yol Haritası

### Faz 0 — Public hazırlık (düşük efor, yüksek görünürlük)
- [ ] Linux header çözümleme düzeltmesi (§1.3a) — **ilk iş; sessiz eksik analiz üretiyor**
- [ ] README (İngilizce: ne, neden, mimari şeması, örnek çıktı, build tarifi)
- [ ] LICENSE (Apache-2.0 önerisi)
- [ ] GitHub Actions CI: Ubuntu 24.04 + LLVM 18, build + ctest
- [ ] Tanı mesajları İngilizce varsayılan, `--lang tr` seçeneği
- [ ] Çapraz-TU bulgu tekilleştirme (§1.3e)

### Faz 1 — Analiz çekirdeğini derinleştirme
- [ ] **Dal koşulu iyileştirmesi (assume edges):** CFG terminator koşulunu
      true/false ardıl kenarlarında state'e işle (`if (p) → p NonNull`,
      `if (z != 0) → z NonZero`). Tek başına en büyük FP sınıfını öldürür
      ve DivByZero'daki bilinen guard FP'sini çözer. **En yüksek
      değer/efor oranlı teknik adım.**
- [ ] UninitPointerRule'u çarpım lattice + dyn_cast desenine taşı (§1.3c)
- [ ] Fonksiyon başına CFG önbelleği (tek build, tüm kurallar paylaşır)
- [ ] İterasyon tavanını lattice yüksekliğine bağla; sabitlenmeyen
      fonksiyonu raporla (§1.3b)
- [ ] `Zero + Unknown` merge kararını guard analiziyle birlikte yeniden ele al (§1.3d)

### Faz 2 — Gerçek dünya kullanılabilirliği
- [ ] SARIF reporter (GitHub code scanning ile bedava entegrasyon)
- [ ] `// zerodefect-disable-line <rule>` bastırma + baseline dosyası
- [ ] Ölçüm altyapısı: NIST Juliet test paketi ile precision/recall
      rakamları — "iddiamızı sayıyla söyleyelim". cJSON/tinyxml2 koşuları
      CI'da regresyon testi olsun.
- [ ] Yeni kurallar (çatı artık ucuzlatıyor): use-after-free (Freed state
      zaten var, dereference kontrolü ekle), null-deref (assume edges
      sonrası anlamlı), taşma-eğilimli döngü sınırları

### Faz 3 — AI döngüsü entegrasyonu (vizyon farklılaşması)
- [ ] Artımlı mod: tek fonksiyon / değişen fonksiyonlar analizi (diff'ten)
- [ ] Bulgulara dataflow izi ekle: "p, 12. satırda alloc → 15'te if-false
      yolu → 19'da tekrar free" — LLM'in tüketeceği açıklama formatı
- [ ] MCP server / JSON-RPC modu: Claude Code ve benzeri ajanların her
      edit sonrası çağıracağı kalıcı süreç (AST cache sıcak kalır)
- [ ] Diff-farkında semantik özet karşılaştırması (§2.3)

### Faz 4 — Araştırma ufku
- [ ] Fonksiyon özetleri ile interprosedürel analiz (sahiplik/nullability
      özetleri — Escaped muhafazakârlığını gerçek transfer bilgisine çevirir)
- [ ] Hafif sözleşme/annotation dili + doğrulayıcı (§2.2)
- [ ] Path-sensitivity'ye kademeli geçiş (önce sadece guard'lı bölgelerde)

### Sıralama gerekçesi
Faz 0 güven inşa eder (public'e açılabilir durum), Faz 1 aracın "oyuncak
değil" eşiğini geçirir (guard analizi olmadan hiçbir gerçek projede sinyal/
gürültü oranı tutmaz), Faz 2 ölçülebilirlik ve adaptasyon getirir, Faz 3
ise kimsenin oturmadığı koltuğa oturur. Faz 3'ü öne çekme cazibesine karşı
not: izi olmayan, FP'si yüksek bir aracı döngüye sokmak LLM'i de yanıltır —
çekirdek kalite (Faz 1) vizyonun ön şartıdır.

## 4. Bu analizde kullanılan Linux build tarifi

```bash
apt-get install -y llvm-18-dev libclang-18-dev libzstd-dev zlib1g-dev
cmake -B build -G Ninja -DCMAKE_PREFIX_PATH=/usr/lib/llvm-18 -DCMAKE_BUILD_TYPE=Release
cmake --build build
ctest --test-dir build          # 41/41
```

Not: `libzstd-dev` olmadan LLVM 18'in CMake export'ları
`zstd::libzstd_shared` hedefini bulamayıp configure aşamasında patlıyor.
CI workflow'u yazılırken bu tarif temel alınabilir; `CMAKE_PREFIX_PATH`
Homebrew varsayılanı yalnızca `APPLE` koşuluna alınmalı.
